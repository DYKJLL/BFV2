### 3.2 P1: LRU BrowserView 内存池 (v2.0)

#### 问题诊断

**原始实现**: 使用无限大小的 `Map` 缓存 BrowserView 实例  
**问题**: 
- 用户频繁切换模块时，Map 无限增长
- 每个 BrowserView 占用 ~50-100MB 内存
- 长时间使用导致内存耗尽，最终 OOM 崩溃

#### 解决方案

实现 LRU (Least Recently Used) 缓存淘汰策略：
- 最大容量: **3个** BrowserView 实例
- 超出时自动淘汰最久未使用的视图
- 淘汰前安全销毁 (三重空值检查 + try-catch)

#### 代码位置

**文件**: [`main.js`](file:///E:\ST\BFV2-Optimization\BFV2-main\main.js#L185-L242)

```javascript
// 第186行: 池大小限制
const MAX_VIEW_POOL_SIZE = 3;

// 第199-221行: LRU淘汰算法
function evictLRU() {
  let oldestUrl = null;
  let oldestTime = Infinity;
  
  for (const [url, viewData] of viewPool.entries()) {
    if (viewData.lastAccessed < oldestTime) {
      oldestTime = viewData.lastAccessed;
      oldestUrl = url;
    }
  }
  
  if (oldestUrl) {
    const oldView = viewPool.get(oldestUrl).instance;
    viewPool.delete(oldestUrl);
    
    // 三重安全检查
    if (oldView && oldView.webContents && typeof oldView.webContents.isDestroyed === 'function') {
      if (!oldView.webContents.isDestroyed()) {
        try { oldView.destroy(); } catch (err) {}
      }
    }
  }
}

// 第223-237行: 添加到池 (带自动淘汰)
function addToPool(url, viewInstance) {
  while (viewPool.size >= MAX_VIEW_POOL_SIZE) {
    evictLRU();
  }
  viewPool.set(url, { instance: viewInstance, lastAccessed: Date.now() });
}
```

**集成点**: 所有原先直接使用 `viewPool.set()` 的地方改为 `addToPool()`:
- [第327行](file:///E:\ST\BFV2-Optimization\BFV2-main\main.js#L327): 预加载时
- [第674行](file:///E:\ST\BFV2-Optimization\BFV2-main\main.js#L674): navigate 处理器
- [第686行](file:///E:\ST\BFV2-Optimization\BFV2-main\main.js#L686): navigate 新建视图
- [第760行](file:///E:\ST\BFV2-Optimization\BFV2-main\main.js#L760): reset-module 处理器
- [第770行](file:///E:\ST\BFV2-Optimization\BFV2-main\main.js#L770): reset-module 新建视图

**退出清理**: 在 `window-all-closed` 事件中添加池清理逻辑。

### 3.3 P2: MutationObserver 安全修复

#### 问题诊断

**原始配置**:
```javascript
observer.observe(targetNode, {
  childList: true,
  subtree: true,
  attributes: true  // ← 问题根源
});
```

**风险**: `attributes: true` 会监听所有属性变化（style、class、id等），在动态页面中可能触发大量回调，形成:
- 回调 → DOM修改 → 触发新回调 → ... (无限循环)
- 导致 CPU 100% 和界面卡死

#### 解决方案

**文件**: [`renderer.js`](file:///E:\ST\BFV2-Optimization\BFV2-main\assets\js\renderer.js#L873-L920)

```javascript
// 移除 attributes: true
// 添加 re-entry lock 防止重入
let isObserving = false;

const observer = new MutationObserver((mutations) => {
  if (isObserving) return;  // 重入锁
  isObserving = true;
  
  requestAnimationFrame(() => {
    // ... 处理逻辑
    isObserving = false;
  });
});

observer.observe(sidebarContent, {
  childList: true,
  subtree: true
  // attributes: true 已移除!
});
```

### 3.4 P3: Event Delegation 优化

#### 问题诊断

**原始实现**: 每次 `refreshDramaSidebar()` 都为新按钮添加独立的事件监听器:

```javascript
// ❌ 问题代码模式
function refreshDramaSidebar() {
  sidebarContent.innerHTML = generateSidebarHTML();
  
  document.querySelectorAll('.drama-btn').forEach(btn => {
    btn.addEventListener('click', handleClick);  // 每次都新增！
  });
}
```

**问题**: 
- 每次侧边栏刷新都累积新的监听器
- 旧监听器无法移除 (匿名函数引用丢失)
- 内存泄漏 + 重复触发

#### 解决方案

**文件**: [`renderer.js`](file:///E:\ST\BFV2-Optimization\BFV2-main\assets\js\renderer.js#L701-L750)

```javascript
// ✅ 一次性容器级委托
sidebarContent.addEventListener('click', async (event) => {
  const btn = event.target.closest('[data-site-url]');
  if (!btn) return;
  
  const url = btn.dataset.siteUrl;
  const name = btn.dataset.siteName || '未知站点';
  
  await handleDramaSiteClick(name, url);
}, { passive: true });
```

**优势**:
- 只注册一个监听器，永不泄漏
- 自动处理动态添加的按钮
- 性能更优 (事件冒泡 vs N个监听器)

### 3.5 BUG FIX: evictLRU TypeError 崩溃

#### 错误现象

用户报告: 点击"国内解析"、"哔哩哔哩"等平台时弹出错误对话框:
```
TypeError: Cannot read properties of undefined (reading 'isDestroyed')
at evictLRU (main.js:xxx)
```

#### 根因分析

```javascript
// 原始代码 (有bug)
function evictLRU() {
  // ...
  const oldView = viewPool.get(oldestUrl).instance;
  // oldView.webContents 可能是 undefined!
  if (!oldView.webContents.isDestroyed()) {  // 💥 CRASH
    oldView.destroy();
  }
}
```

当 BrowserView 被 Electron 内部回收但 Map 引用未更新时，`webContents` 为 `undefined`。

#### 修复方案

```javascript
// 修复后的代码 (三重检查)
if (oldView && oldView.webContents && typeof oldView.webContents.isDestroyed === 'function') {
  if (!oldView.webContents.isDestroyed()) {
    try { oldView.destroy(); } catch (err) {}  // 额外try-catch
  }
}
```

**验证结果**: 用户确认 "目前国内解析模块下的这几个播放源都是正常的" ✅

const isHeavySite = url.includes('movie1080') || url.includes('monkey-flix');

### 3.9 模块切换功能尝试 (已回退)

#### 需求

用户要求: "实现模块切换功能...1.立即结束当前任务 2.刷新网页 3.自动导航至目标URL"

#### 实现

编写了 `smoothModuleSwitch()` 函数 (217行)，包含三个阶段:

```
Phase 1: Immediate Task Termination
  ├── view.webContents.stop()
  ├── setAudioMuted(true)
  ├── removeBrowserView(view)
  ├── clearHistory()
  └── executeJavaScript(DOM清理)

Phase 2: Prepare New View
  ├── 检查 LRU Pool 是否有缓存
  ├── 有: 复用现有视图
  └── 无: 创建新视图

Phase 3: Navigate to Target URL
  ├── 注入主题CSS
  ├── 设置缩放因子
  ├── loadURL(targetUrl)
  └── 处理 did-finish-load / did-fail-load
```

#### 结果

**❌ 失败**: 用户反馈 "不行这个版本，直接用不了，所有页面都无法加载出来"

**原因推测**:
- Phase 1 中的 `executeJavaScript` DOM清理过于激进
- 或 `clearHistory()` + `session.clearCache()` 组合破坏了某些状态
- 或异步等待导致时序问题

#### 处理

**完全回退**:
1. 删除 `smoothModuleSwitch()` 函数 (217行)
2. 恢复 `navigate` handler 到 v2.5 稳定版
3. 恢复 `reset-module` handler 到 v2.5 稳定版
4. 重新打包 V1.2.2

**结论**: 当前 v2.5 的处理器已经足够稳定，无需过度复杂的切换机制。

## 7. 技术架构说明

### 7.1 数据流架构

```
用户点击模块按钮
    ↓
Renderer Process (renderer.js)
    ↓ ipcRenderer.send('reset-module', url)
Main Process (main.js)
    ↓
┌─────────────────────────────────────┐
│ reset-module handler                │
│ 1. view.webContents.stop()          │
│ 2. view.webContents.setAudioMuted() │
│ 3. mainWindow.removeBrowserView()   │
│ 4. addToPool(currentUrl, view)      │ ← LRU池管理
│ 5. view = createNewBrowserView()    │  或 get from pool
│ 6. addToPool(url, view)             │
│ 7. mainWindow.setBrowserView(view)  │
│ 8. view.webContents.loadURL(url)    │
└─────────────────────────────────────┘
    ↓ did-finish-load
injectThemeCss() + updateZoomFactor()
    ↓
ipcRenderer.send('load-finished') → UI更新
```

### 7.2 LRU Pool 工作流程

```
初始状态: Pool = {}

用户操作: 点击 腾讯视频
  → Pool = { "v.qq.com": { instance: View1, time: T1 } }

用户操作: 点击 爱奇艺
  → Pool = { "v.qq.com": View1, "iqiyi.com": { instance: View2, time: T2 } }

用户操作: 点击 优酷
  → Pool = { "v.qq.com": View1, "iqiyi.com": View2, "youku.com": { instance: View3, time: T3 } }
  → Size = 3 = MAX ✓

用户操作: 点击 哔哩哔哩 (需要新建)
  → Size would be 4 > MAX=3
  → evictLRU() 淘汰最旧的 (T1: v.qq.com)
  → destroy(View1) 安全释放
  → Pool = { "iqiyi.com": View2, "youku.com": View3, "bilibili.com": { instance: View4, time: T4 } }

用户操作: 再次点击 腾讯视频
  → 不在 Pool 中 → 创建 View5
  → evictLRU() 淘汰 T2: iqiyi.com
  → Pool = { "youku.com": View3, "bilibili.com": View4, "v.qq.com": { instance: View5, time: T5 } }
```

### 7.3 Settings 数据流

```
Renderer                    Main                      File System
   │                          │                            │
   ├─ getAllSettings() ──►   │                            │
   │                          ├─ settingsStore.store       │
   │                          │    (读取 JSON 文件) ──────►│ userData/user-settings.json
   │◄── { success, data } ──┤                            │
   │                          │                            │
   ├─ setSetting(key,val)─►  │                            │
   │                          ├─ settingsStore.set()       │
   │                          │    (写入 JSON 文件) ──────►│ userData/user-settings.json
   │◄── { success } ────────┤                            │
```
